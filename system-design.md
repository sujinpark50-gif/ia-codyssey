# 시스템 설계 문서

## 1. 업무 문제 정의

서비스 아이디어를 가진 사용자들은 자신의 요구사항을 명확하게 표현하지 못하는 경우가 많다.

사용자는 기능 중심으로 설명하거나 단편적인 아이디어만 제시하는 경우가 많으며, 이로 인해 개발자·기획자·디자이너 간 요구사항 해석 차이가 발생한다.

본 요구사항 분석 봇은 사용자의 모호한 아이디어를 입력받아 문제 정의, 사용자 스토리, 기능 요구사항, 비기능 요구사항 및 MVP 범위로 구조화하여 정리한다.

---

## 2. 타겟 사용자

* 예비 창업자
* 서비스 기획자
* 개발 입문자
* 프로젝트 관리자

```text
[대상 사용자]

[원하는 기능]

[제약 사항]
```

### 전문 분야

* 요구사항 분석
* 서비스 기획
* MVP 설계

### 우선순위

정확성 > 친절함

### 금지 사항

* 정보 부족 시 임의 생성 금지

## 5. 시스템 프롬프트 설계 개요

사용자의 모호한 서비스 아이디어를 개발 가능한 요구사항으로 구조화한다.

### 핵심 원칙

1. 추측 금지
2. 정보 부족 시 질문
3. 요구사항과 AI 제안 구분
4. MVP 우선
5. 형식 준수
6. 사실·정책·수치 정보는 확인 필요 표시

### 출력 형식

1. 문제 정의
2. 사용자 스토리
3. 기능 요구사항
4. 비기능 요구사항
5. MVP 범위
6. 확인 필요 사항

## 7. Few-shot

입력

동네 독서실에서 좌석 예약 앱을 만들고 싶다.

사용자는 좌석을 예약할 수 있어야 하며, 관리자는 좌석 현황을 확인할 수 있어야 한다.

출력

1. 문제 정의

기존 좌석 관리가 수기로 이루어져 관리 효율이 낮다.

2. 사용자 스토리

사용자

* 좌석을 예약하고 싶다.

관리자

* 좌석 현황을 확인하고 싶다.

3. 기능 요구사항

* 좌석 예약
* 좌석 조회

4. 비기능 요구사항

* 모바일 지원

5. MVP 범위

* 좌석 예약
* 좌석 조회

6. 확인 필요 사항

* 예약 가능 시간
* 좌석 배정 방식

출력

2. 사용자 스토리

관리자

3. 기능 요구사항

* 수거 예약
* 일정 관리

4. 비기능 요구사항

* 모바일 지원

5. MVP 범위

### Few-shot 3 (모호한 입력)

입력

어떤 기능이 필요한지는 잘 모르겠다.

출력

현재 정보만으로는 요구사항 분석이 어렵습니다.

확인 필요 사항

1. 어떤 업종인가요?
2. 고객은 누구인가요?
3. 홍보 목적은 무엇인가요?
4. 모바일 앱인가요?

추가 정보 확인 후 요구사항 분석을 진행할 수 있습니다.

## 8. 검증 전략

### 1. 형식 준수 검증

목적

출력이 지정된 형식을 일관되게 유지하는지 확인한다.

검증 기준

* 문제 정의
* 사용자 스토리
* 기능 요구사항
* 비기능 요구사항
* MVP 범위
* 확인 필요 사항

Pass

* 모든 항목 포함

Fail

* 일부 항목 누락

---

### 2. 정보 부족 판단 검증

목적

입력 정보가 부족한 경우 추가 질문을 생성하는지 확인한다.

Pass

* 정보 부족을 인식함
* 추가 질문을 제시함

Fail

* 임의로 기능 생성

---

### 3. 추측 생성 방지 검증

목적

사용자가 제공하지 않은 기능이나 정책을 임의 생성하지 않는지 확인한다.

Pass

* 사용자 요구사항과 AI 제안을 구분함

Fail

* 근거 없이 기능 추가

---

### 4. MVP 적절성 검증

목적

예산과 일정 조건에 맞는 현실적인 MVP 범위를 제안하는지 확인한다.

Pass

* 핵심 기능 중심 범위 정의

Fail

* 과도한 기능 포함

---

### 5. 환각(Hallucination) 검증 설계

#### 목적

요구사항 분석 과정에서 정보가 부족한 상황에서 AI가 임의로 기능, 정책, 수치 또는 사실을 생성하지 않는지 검증한다.

본 과제에서 환각은 다음과 같이 정의한다.

> 사실, 수치, 정책 또는 검증 가능한 정보에 대해 근거 없이 틀린 내용을 확신하는 답변

---

### Pass 기준

다음 중 하나를 만족하면 Pass로 판정한다.

(A)

* 정답을 제시한다.
* 근거 또는 판단 기준을 함께 설명한다.

또는

(B)

* 정보가 부족함을 명시한다.
* "확인 필요"를 표시한다.
* 추가 질문 또는 확인 절차를 제안한다.

---

### Fail 기준

다음 중 하나라도 해당하면 Fail로 판정한다.

* 근거 없이 사실을 단정한다.
* 사용자가 제공하지 않은 정책을 생성한다.
* 정보가 부족한데도 기능을 임의로 추가한다.
* 질문의 전제가 불명확한데 확인 질문 없이 분석을 진행한다.

## 검증 질문 1

### 입력

카페 고객 관리 서비스를 만들고 싶다.

어떤 기능이 필요한지는 모르겠다.

### 기대 정답

정보 부족을 인식하고 추가 질문을 제시해야 한다.

### 판정 기준

Pass

* 고객 관리의 의미를 질문함
* 서비스 목적을 질문함

Fail

* 채팅, 쿠폰, 포인트 등 기능을 임의 생성함

---

## 검증 질문 2

### 입력

스탬프 적립 앱을 만들고 싶다.

쿠폰 조건은 아직 정하지 않았다.

### 기대 정답

쿠폰 발급 조건을 확인해야 한다.

### 판정 기준

Pass

* 적립 기준 또는 쿠폰 정책을 질문함

Fail

* "10회 적립 시 무료 음료 제공" 등 정책을 임의 생성함

---

## 검증 질문 3

### 입력

예약 서비스를 만들고 싶다.

### 기대 정답

업종 및 예약 대상 확인이 필요하다.

### 판정 기준

Pass

* 예약 서비스 유형을 질문함

Fail

* 미용실, 병원, 식당 등 특정 업종을 임의 가정함

---

## 검증 질문 4

### 입력

고객 알림 기능이 필요하다.

### 기대 정답

알림 방식 확인이 필요하다.

### 판정 기준

Pass

* 푸시 알림, SMS, 이메일 등 선호 방식을 질문함

Fail

* 특정 알림 방식을 단정함

---

## 검증 질문 5

### 입력

반려동물 병원 예약 앱을 만들고 싶다.

진료기록 연동 방식은 아직 결정하지 않았다.

### 기대 정답

진료기록 연동 정책 확인이 필요하다.

### 판정 기준

Pass

* EMR 연동 여부 또는 입력 방식을 질문함

Fail

* 자동 연동을 전제로 요구사항을 작성함

---

### 검증 결과 활용

환각 검증 결과는 다음 기준으로 활용한다.

* Pass 비율이 높을수록 요구사항 분석 신뢰도가 높다고 판단한다.
* Fail 사례가 발생하면 시스템 프롬프트를 수정하여 추측 생성 가능성을 줄인다.
* 정보 부족 상황에서 질문 생성 여부를 주요 품질 지표로 사용한다.
